# Landing page visual upgrade with framer-motion animations and glassmorphism

The landing page is wrapped in a suite of client-side animation components: an animated gradient background with 30 floating particles, 3D rotating shapes, glassmorphism feature cards with mouse-tracking tilt, scroll-triggered reveals, counting-number stats, and ripple-effect buttons. The page itself remains a server component that imports these client leaves, and framer-motion is the sole new dependency. Build and lint pass.

**Watch for:** 30 continuously-animating particles plus rotating cubes and orbiting circles create a significant GPU workload with no reduced-motion escape hatch (confirmed). The `AnimatedButton` wrapper intercepts clicks on a `<div>` around a `<Link>`, which may swallow navigation on certain mobile browsers and creates a non-semantic interactive element (likely). Ripple state in `AnimatedButton` grows unboundedly for rapid clickers until the 600ms timeout catches up (confirmed, minor).

## High-level view

`page.tsx` stays as a server component and delegates all interactivity to six new `'use client'` leaf components under `src/components/landing/`. The risk is purely on the client side, where 30+ concurrently animating DOM nodes (particles, cubes, circles, lines, glowing dots) run infinite animation loops with no `prefers-reduced-motion` check anywhere, meaning users with vestibular sensitivities or low-end devices get the full animation load with no opt-out.

The `AnimatedButton` component wraps an `<a>` (via `<Link>`) inside a motion `<div>` with `onClick`. This creates a semantic accessibility gap and a potential navigation interception problem on middle-click, ctrl+click, and keyboard interaction.

The navbar's new `bg-white/80 backdrop-blur-lg` introduces transparency that may reduce text contrast below WCAG AA against colorful page content, and its 600ms slide-down entrance delays user access to navigation links.

<summary>Issues (7)</summary>

1. **No reduced-motion support** — Add a `prefers-reduced-motion` media query check (via `useReducedMotion` from framer-motion or a CSS media query) to disable or simplify animations for accessibility compliance (WCAG 2.1 SC 2.3.3).
2. **AnimatedButton wraps Link in a div with onClick** — The outer `<div>` intercepts click events before the `<Link>` `<a>` tag, which can interfere with native link behavior (middle-click, ctrl+click, long-press). Consider moving the ripple logic into a styled `<a>` or using framer-motion's `motion(Link)`.
3. **AnimatedButton is not a semantic interactive element** — A `<div>` with `whileHover`/`whileTap` doesn't receive keyboard focus or announce itself to screen readers. The inner `<Link>` is focusable, but the scale animation won't trigger on keyboard interaction.
4. **Heavy concurrent animation count** — 30 particles + 3 rotating cubes + 3 orbiting circles + 3 connecting lines + 3 glowing dots = 42 simultaneously animating nodes, all with `repeat: Infinity`. On lower-end mobile GPUs this may cause jank or battery drain. Reduce particle count or use CSS-only animations for particles.
5. **Ripple array grows without bound during rapid clicks** — Each click pushes to state immediately; cleanup fires 600ms later. A burst of 20 clicks creates 20 simultaneously rendering `<span>` elements. Cap the array or use a single-ripple model.
6. **Counter displays nonsensical intermediate values** — "0K+" through "10K+" reads oddly; consider counting to 10000 and formatting dynamically, or fading in the final value for suffixed numbers.
7. **Navbar contrast and interaction delay** — `bg-white/80` over dark or colorful content may drop nav link contrast below WCAG AA. The 600ms entrance animation delays access to navigation links.

## Accessibility gap: no reduced-motion path

None of the six animation components check `prefers-reduced-motion`. Framer-motion exposes `useReducedMotion()` that returns `true` when the OS preference is set; the components could early-return static markup or pass `animate={false}` to their motion elements. The CSS keyframe animations in `globals.css` similarly lack a `@media (prefers-reduced-motion: reduce)` block to disable them. This is a WCAG 2.1 SC 2.3.3 violation for motion-triggered content.

```css
/* Missing from globals.css */
@media (prefers-reduced-motion: reduce) {
  .animate-gradient-shift,
  .animate-float,
  .animate-pulse-glow,
  .animate-particle-drift,
  .animate-ripple {
    animation: none;
  }
}
```

## AnimatedButton semantics and navigation interception

`AnimatedButton` renders a `motion.div` with `onClick`, `whileHover`, and `whileTap`. Inside it sits a `<Link>` which renders an `<a>`. The layering means:

- Middle-click or ctrl+click on the outer div doesn't trigger `<a>` navigation behavior. The browser sees the click target as a `<div>`, not a link.
- The `scale` transforms on `whileHover`/`whileTap` are bound to the `<div>`'s pointer events; keyboard focus on the `<a>` inside won't trigger them.
- The ripple `<span>` elements are `pointer-events-none`, which is correct, but the wrapping `overflow-hidden` clips the `<Link>`'s focus ring.

A cleaner pattern would use `motion(Link)` directly, or attach the ripple logic to the `<a>` element itself.

## GPU budget for the hero section

The hero section stacks three layers of animation:

```
HeroBackground (gradient shift + secondary overlay pulse + 30 particles)
FloatingShapes (3 cubes rotating, 3 circles orbiting, 3 lines fading, 3 dots pulsing)
ScrollReveal (3 stagger-delayed entrance animations, one-shot)
```

The entrance animations fire once, but everything else runs indefinitely. Each framer-motion `animate` with `repeat: Infinity` creates a persistent animation frame callback. Thirty particles alone mean 30 independent animation tracks. On mid-range Android devices, this risks dropped frames during initial page load when the entrance animations are also running.

Mitigations: reduce particle count to 10-15, use CSS-only animations (GPU-composited without JS frame callbacks) for the particles, or pause off-screen animations using Intersection Observer.

## Counter animation edge case

`AnimatedCounter` parses `"10K+"` into `{ num: 10, suffix: "K+" }` and counts from 0 to 10, displaying "0K+" through "10K+". The early frames showing "0K+" and "1K+" are nonsensical since you can't have "zero thousand" of something. A more polished approach would count to 10000 and format with a suffix dynamically, or fade-in the final value for non-integer suffixes.

## Navbar entrance animation delays interaction

The navbar animates from `y: -100, opacity: 0` to `y: 0, opacity: 1` over 600ms. During this window, navigation links are invisible and effectively unreachable. On slow connections where the JS bundle takes time to hydrate, this combines with hydration delay to extend the no-nav period. The `bg-white/80` transparency also means that when a user scrolls the hero gradient under the navbar, `text-gray-700` nav links sit against a blended purple/blue background, potentially failing WCAG AA contrast (4.5:1 ratio) for the lighter gray text.
